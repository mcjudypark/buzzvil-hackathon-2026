💡 쉽게 말하면
회사 서버를 집 조명에 비유하면, 지금은 아무도 없어도 24시간 불을 켜두는 상태예요. 실제로 쓰는 시간은 하루 중 21%뿐인데 돈은 계속 나가고 있었죠. 이 팀은 "아무도 없으면 끄고, 누가 접속하면 자동으로 켜는" 시스템을 만들어 비용을 92%나 줄였어요. 심지어 개발자 51명 전원에게 각자의 개발 공간을 줘도 현재보다 훨씬 저렴해요.
현재 문제 (수치로)
• 개발·테스트 서버 90개가 24시간 가동 중
• 실제 사용률: 하루 평균 5.1시간 (21.5%)만 사용, 나머지 78.5%는 유휴
• 월 비용: 개발 서버 예산의 대부분이 유휴 상태에 낭비되는 구조
• 지난달(3월) 슬랙에서 "staging 쓰시나요?", "배포해도 될까요?" 같은 환경 충돌 메시지 12건 발생
해결 방법
• 배포하면 서버 자동 기동, 30분간 요청 없으면 자동 종료
• 의존하는 다른 서비스들도 함께 자동으로 켜지고 꺼짐 (17개 서비스 의존관계 전수 조사 완료)
• 기존 배포 시스템은 그대로, 슬립/웨이크 레이어만 추가 → 개발자 경험 변화 없음
• 개발자 51명 × 15개 서비스 = 765개 독립 환경도 비용 부담 없이 가능
📊 비용 시나리오 비교
• 현재 (90개 공유, 상시 가동): 기준 비용 (100%)
• 현재 서버 + 슬립 적용: 81.8% 절감
• 개발자별 독립 환경 + 슬립: 92.4% 절감 (최대 효과)
• 개발자별 환경을 상시 가동하면: 비용 6배 증가 → 슬립 적용이 핵심

💡 쉽게 말하면
새로 입사하거나 다른 팀과 협업할 때 "이건 누구한테 물어봐야 하지?", "이 결정이 왜 이렇게 된 거지?" 같은 질문을 Slack과 노션을 뒤지며 40~50분씩 쓰게 돼요. 이 팀은 회사의 모든 기록을 AI가 이해할 수 있게 정리해두고, 그냥 물어보면 바로 답해주는 사내 AI 비서를 만들었어요. 기존 유료 사내 검색 솔루션 대비 라이선스 비용도 대폭 절약돼요.
현재 문제
• 버즈빌리언 128명이 정보 탐색에 하루 40~50분 소비
• 정보가 Slack·Confluence·Linear에 파편화 — 어디에 있는지조차 모르는 상황
• 기존 MCP+Claude Code는 정제 안 된 원시 데이터에 바로 접근 → 노이즈 과다
• 신규 입사자 온보딩, 타 팀 협업 시 맥락 파악에 특히 어려움
EAS 모델 — 핵심 구조
사내 데이터를 3가지로 분류해 AI가 이해할 수 있게 정제:
• Entity (누가/무엇이): 사람 49명 · 팀 33개 · 시스템·제품 15개
• Activity (무슨 일이): 토론 · 결정 · 작업 · 업데이트 — 현재 436개 적재
• Source (어디서, 신뢰도): Confluence(높음) → Linear(중간) → Slack(낮음) 계층 구조
현재 데이터 현황

• Slack 35개 채널 → 330개 Activity 추출

Vision Decomposer
💡 쉽게 말하면
"올해 우리 팀 목표는 X입니다"라는 비전이 있어도, 실무자는 "그래서 내가 월요일에 뭘 해야 하지?"를 모를 때가 많아요. 이 팀은 큰 목표를 AI가 구체적인 할 일로 자동으로 쪼개주고, 다른 AI가 "이게 맞게 쪼갠 건지" 검토해주는 도구를 만들었어요. 개발팀이 코드를 PR로 리뷰하듯, 전략도 같은 방식으로 합의해요.
핵심 문제
• AI 시대의 진짜 병목은 "어떻게"가 아닌 "무엇을 해야 하는가"
• 비전 → 전략 → 과제로 내려갈수록 맥락이 사라져 실무자는 "왜 이걸 하는지" 모른 채 일함
• 이해관계자마다 비전 해석이 달라 정렬 실패 반복
• AI에게 과제를 위임하려면 충분한 맥락이 담긴 단위로 분해되어야 함
작동 방식 (5단계)
① 비전과 맥락을 마크다운 문서로 작성
② 분해 AI가 실행 가능한 단위 과제로 트리 구조로 분해
③ 검증 AI (다른 모델)가 분해 품질 검토 — CI처럼 자동 실행
④ 팀원들이 트리 diff를 PR처럼 리뷰하고 합의
⑤ 각 Leaf 노드(최소 과제)엔 배경·범위·완료 기준 포함 → AI 에이전트에 바로 위임 가능
